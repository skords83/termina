## Anti-Patterns Verdict

**Start here.** Does this look AI-generated? **Partial** — and the two assessments agree with each other in a load-bearing way.

**LLM assessment**: The calendar-grid CSS (`index.css`, `layout.css`, `calendar.css`, `views.css`, `dnd.css`) is genuinely disciplined — no gradient text, no glassmorphism, no hero-metric tiles, no eyebrow/numbered scaffolding, no decorative side-stripes beyond the sanctioned per-calendar event border. It reads like a real execution of "The Warm Signal," not a template. But `EventFormModal.tsx` and `EventPopup.tsx` are built with hardcoded inline JS style objects using an off-brand palette (`#5b8ef7` instead of the documented Signal Blue `#4f7ef8`; `#1e1e1e`/`#2e2e2e` instead of `--bg-2`/`--border`). `EventFormModal.tsx` (~line 1110) renders a literal decorative gradient bar — `linear-gradient(90deg, #5b8ef7, #7c5bf7)` — introducing an undocumented second hue (purple) at the top of the "New Event" modal. That's a direct violation of DESIGN.md's One Signal Rule and close to a textbook AI-slop tell.

**Deterministic scan**: `detect.mjs` returned 109 findings against `frontend/src` (exit code 2): 75 `design-system-color`, 31 `design-system-radius`, 3 `side-tab` (all `warning` severity, rest `advisory`). Findings concentrate almost exactly where the LLM review flagged the problem by reading the code independently: `EventFormModal.tsx` alone accounts for 44 findings (36 color + 8 radius), `EventPopup.tsx` for 15. That's strong agreement between a mechanical scan and a qualitative read — the two-design-system problem is real, not a stylistic nitpick.

The 3 `side-tab` findings (`calendar.css:240`, `views.css:159`, `views.css:369`) are **false positives relative to this project's own system** — they're the `border-left` on event chips using `var(--event-color)`, which DESIGN.md explicitly documents as the intentional, functional Calendar-Owns-Its-Color exception, not a decorative accent. Correctly flagged by the detector (it has no way to know about your exception), correctly dismissed here.

**Live browser evidence**: The backend isn't running in this environment, so only the login/token-gate screen was reachable. The detector ran directly in that rendered page and found 3 real issues: text `#5c6070` on `#16181d` (`p.login-sub`) at **2.8:1** contrast (needs 4.5:1), white button text on `#4f7ef8` (`button.login-btn`) at **3.7:1** (needs 4.5:1), and a flat type hierarchy on the page (13/14/15/18px, only a 1.4:1 ratio). This is a genuine WCAG AA failure on the one screen every session opens with — this was an isolated agent-controlled browser session, not a tab left open for you to inspect directly.

## Overall Impression

The calendar grid itself is a well-executed, disciplined instance of "The Warm Signal" — restrained, Signal-Blue-only, flat by default. The gap between that and the event-creation flow (the single most-used interaction in the app) is the biggest opportunity here: `EventFormModal`/`EventPopup` look like they were built in an earlier pass, before the design system settled, and never reconciled. Fixing that one gap would move the aesthetic/consistency scores up substantially without touching anything that's currently working.

## What's Working

- **The calendar grid CSS** (`layout.css`, `calendar.css`, `views.css`) faithfully implements the neutral ramp + One Signal Rule — today's date, drop targets, and the now-line are the only saturated blue in the whole grid.
- **Recurring move/delete dialogs** (`RecurringMoveDialog.tsx`, `EventPopup.tsx`'s delete-scope dialog) give plain-language, option-specific consequences — exactly the reassurance a high-stakes irreversible action needs.
- **`NaturalInputBar.tsx`** does progressive disclosure right: contextual example hints when empty, a live preview once parseable, an explicit fallback warning when not — all inline, no nested modal.

## Priority Issues

**[P1] Two divergent design systems, including an off-brand gradient bar in the primary event-creation flow**
- **Why it matters**: `EventFormModal`/`EventPopup` are the most-used surfaces in a "fast glance-and-plan" tool, and they visually contradict the rest of the app — a different blue, different neutrals, and a second hue (purple gradient) that violates the One Signal Rule you just wrote down in DESIGN.md. Confirmed independently by both the design review and the detector's 44+15 findings concentrated in exactly these two files.
- **Fix**: Delete the inline `S`/`dpS` style objects in `EventFormModal.tsx`/`EventPopup.tsx`, reuse the existing `.popup`/`.rec-dialog` classes already defined in `modals.css`, and remove the gradient bar entirely.
- **Suggested command**: `/impeccable polish frontend/src/components/EventFormModal.tsx`

**[P1] No visible keyboard focus indicator anywhere except the login screen**
- **Why it matters**: `EventFormModal`'s inline inputs explicitly set `outline: "none"` with nothing to replace it, and `:focus` is defined exactly once in all six CSS files (on `.login-input`). A keyboard-only user cannot see where focus is inside the event form.
- **Fix**: Add a global `:focus-visible` rule using the system's existing color-shift language (border → Signal Blue, matching the input focus treatment DESIGN.md already documents) and remove the `outline: none` overrides.
- **Suggested command**: `/impeccable audit frontend/src/components/EventFormModal.tsx`

**[P1] Silent read-path failures**
- **Why it matters**: `useCalendars`/`useEvents` both return an `error` value that `App.tsx` never destructures or displays. If the backend is unreachable, the app renders an empty, unexplained calendar instead of telling the owner anything is wrong — directly under cutting Heuristic 1 (Visibility of System Status).
- **Fix**: Surface `error` via the existing `showToast` system or a persistent banner with retry, matching the specificity already used for write-path errors.
- **Suggested command**: `/impeccable harden frontend/src/App.tsx`

**[P1] Recurring "delete entire series" is one click despite being labeled irrevocable**
- **Why it matters**: The copy calls the action "unwiderruflich" (irrevocable), but it fires in a single click — inconsistent with the two-step confirm→"Wirklich löschen" flow already built for single-event delete in the same file. This is the highest-stakes destructive action in the app and it has the weakest guardrail.
- **Fix**: Add a matching second confirmation step, or an undo-window toast, for the "all" scope specifically.
- **Suggested command**: `/impeccable harden frontend/src/components/EventPopup.tsx`

**[P1] Login screen fails WCAG AA contrast, and it's the screen every session opens with**
- **Why it matters**: Confirmed live by the detector in-browser: `p.login-sub` at 2.8:1 and `button.login-btn` text at 3.7:1, both below the 4.5:1 body-text minimum. Compounded by PRODUCT.md's own admission that the login screen already reads "closer to a precise developer tool than warm" — the contrast failure makes it also read as unpolished, not just cold.
- **Fix**: Darken/brighten `--text-3` for use on `.login-sub` specifically (or promote that text to `--text-2`), and either lighten the button background or use a darker text color on `.login-btn` to clear 4.5:1.
- **Suggested command**: `/impeccable audit frontend/src/components/LoginForm.tsx`

## Persona Red Flags

**Alex (Impatient Power User)**: The entire shortcut system (`n`, `t`, `1`-`4`, space, ⌘K, ⌘N) is undiscoverable beyond trial-and-error or reading source — no `?` help overlay exists. The natural-language parser silently defaults unparseable text to "heute 09:00" with only a small inline note; firing off ⌘N → type → Enter without reading closely could commit an event at the wrong time.

**Sam (Accessibility-Dependent User)**: The missing focus-visible styling (Priority Issue above) makes keyboard-only navigation of the event form effectively blind — you can't see which field has focus. `Toast.tsx`'s toast item is a `div` with `onClick`, not a button, with no `role="status"`/`aria-live`, so screen readers won't announce save/delete confirmations. And the live-confirmed 2.8:1 / 3.7:1 contrast failures on the login screen are a real WCAG AA blocker on the very first screen Sam would encounter.

**Casey (Distracted Mobile User)**: Zero `@media` queries exist across all six CSS files, despite `TouchSensor` being explicitly configured for drag-and-drop (200ms delay/8px tolerance) — mobile *interaction* was considered, mobile *layout* was not. The fixed 220px sidebar and 52px topbar would consume a large fraction of a phone viewport with no adaptation.

## Minor Observations

- `week-now-line`/`day-now-line` fall back to `var(--accent, #e05a4b)` — a coral/orange fallback from what looks like an earlier palette. Harmless since `--accent` is always defined, but worth cleaning up as scaffolding debris.
- `executeMove`'s catch block falls back to a native `alert()` while every other write path uses the styled Toast system — jarring but low-frequency (only on drag-write failure).
- `SearchModal.tsx`'s highlight color (`rgba(255,200,0,0.3)`, gold/yellow) is a third undocumented accent alongside the EventFormModal gradient's purple.
- `EventFormModal`'s date/time row mixes a native `<input type="time">` with a fully custom `DatePicker` in the same field group — a small visible inconsistency in control styling.

## Questions to Consider

- If `EventFormModal`/`EventPopup` were built before the design system settled, should reconciling them happen before or after the `colorize`/`typeset` passes PRODUCT.md anticipates — otherwise those future passes inherit two systems instead of one.
- PRODUCT.md itself flags the login screen as reading cold — is a bearer-token gate even the right entry pattern for a single-owner PWA, or is there room for something whose first sentence isn't developer jargon?
- Was the one-click "delete all" on recurring events a deliberate call, or inherited from the move dialog's pattern without re-examining that delete's stakes are categorically higher?
